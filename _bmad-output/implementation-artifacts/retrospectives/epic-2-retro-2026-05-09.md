## Epic Summary

Epic 2 的 6 个 Story 均已完成，交付了 CORD 的扫描主链路：框架适配器接口与 Generic fallback、remark/unified 扫描管道、BMAD 适配模块、配置加载与扫描范围控制、ScanService 冷启动写库、增量扫描与文档生命周期检测。

本 Epic 将 CORD 从“工程基座”推进到“可建立文档关系图谱”的核心能力阶段。`cord scan` 已具备从配置加载、适配器解析、文档发现、AST 解析、规则关系发现、framework preset 关系生成、去重过滤、事务写库到增量刷新和无变更快返的完整闭环。

Sprint 状态中 Epic 2 的全部 Story 为 `done`，`epic-2-retrospective` 在本回顾后标记为 `done`。需要注意的是，`epic-2` 主状态仍显示 `in-progress`，建议后续由流程 owner 确认是否手动更新为 `done`。

## What Went Well

1. Epic 2 的垂直链路完整打通。Story 2.1-2.6 从 adapter 抽象到 scanner 管道、配置、Service、CLI、Repository 写入和增量扫描逐层落地，没有中途改写产品方向。
2. CR 发现推动了实质性质量提升。典型例子包括 Story 2.1 修复 symlink 越界/循环扫描风险，Story 2.2 将 `ScanPipeline.process()` 自身接入非 Markdown 与超大文件预检，Story 2.5 修复真实 CLI 入口 `parseAsync()`，Story 2.6 修复同 contentHash rename/move 的错误 docId 绑定风险。
3. 测试策略跟随风险扩展。每个阻塞修复都配套了定向回归测试，而不是只做局部实现修补。Story 2.6 尤其值得保留：既覆盖可消歧 rename/move，也覆盖真正歧义时降级为 delete + add。
4. 范围边界处理较稳。Story 2.3 明确 v0.1 仅支持 16 种 Markdown BMAD 文档类型，YAML 类型延后；Story 2.4 明确 IDE 预设路径归 Epic 5；Story 2.5 明确 rebuild 的 manual 关系保护留到 Epic 4；Story 2.6 明确 rename/move 在 v0.1 只更新路径。
5. 规则沉淀质量提高。Story 2.6 的 CR-SCAN-01 已进入 project context，避免后续 agent 用 FIFO、字典序或数组顺序做弱身份重绑定。

## Challenges

1. 文件系统边界反复出现。Story 2.1、2.3、2.6 都暴露了文件系统遍历、路径竞态、symlink、hash 匹配或大目录性能问题，说明扫描类功能的真实风险主要在边界条件，而不是 happy path。
2. 性能验收口径仍有局限。Story 2.6 的无变更快返虽然避免执行 pipeline，但仍在判定前全量读取并计算 contentHash；当前性能测试覆盖小样本，不足以证明大样本口径。
3. 检测器精度仍有技术债。BMAD detector 的 frontmatter 结束标记、候选扫描上限和异常路径容错都进入 P2 TODO，后续如果扩展到真实大型项目，这些问题会更容易浮出水面。
4. CLI 可观测性仍留有旧债。`--verbose` 在 async action 后才生效的问题从 Story 1.2 延续到 Story 2.5，说明入口级选项时序需要一次专门收口。
5. Sprint 状态存在流程口径漂移。所有 Epic 2 Story 已完成，但 `epic-2` 主状态仍为 `in-progress`。这不影响本次 retrospective 产物，但会影响后续状态查询的准确性。

## Key Insights

1. 扫描引擎的质量核心不是“能读 Markdown”，而是“在不可信文件系统和不完美文档结构中保守地产生可解释结果”。
2. 增量扫描中的身份重绑定必须保守。唯一最优匹配和歧义降级比“稳定但任意”的排序更符合用户数据安全。
3. CR TODO 需要按主题聚合，不应只按 Story 分散记录。Epic 2 的待办自然聚为四类：文件系统鲁棒性、扫描性能、检测器精度、CLI 可观测性。
4. Epic 3 可以启动，但需要把“查询建立在已持久化图谱之上”的假设写清楚：查询服务不应重新扫描，也不应绕过 Repository 接口直接读实现细节。
5. 当前 v0.1 范围控制是正确的。未把 IDE adapter、manual relation 保护、YAML 文档类型、路径敏感 preset 刷新强行塞进 Epic 2，降低了主链路交付风险。

## Next Epic Preview

Epic 3 的目标是“关系查询、影响分析与数据导出”，包含 QueryService 一跳查询、多跳遍历、ImpactService、JSON 快照导出和 StatusService 健康检查。

Epic 3 对 Epic 2 的依赖非常直接：

1. QueryService 依赖 Epic 2 已写入 SQLite 的 `documents`、`relations` 和 `sync_states` 数据。
2. 多跳遍历依赖关系边的类型、来源、置信度和端点完整性。
3. ImpactService 依赖 9 种传播行为类型和关系置信度过滤语义。
4. ExportService 依赖图谱快照的稳定 schema 与 Repository 查询接口。
5. StatusService 依赖增量扫描写入的同步状态、mtime/contentHash 和孤立关系检测能力。

Epic 3 当前方向无需重写。需要提前注意的是，Epic 3 不应把 Epic 2 的扫描性能 TODO 混入查询 Story 主线；查询链路应优先验证 Repository 查询语义、输出契约和错误体验。
