## Epic Summary

Epic 5 的 5 个 Story 均已完成，交付了 CORD 的 AI IDE 集成主链路：MCP Server 从占位入口升级为长驻 stdio 服务，核心能力通过 7 个 MCP Tools 暴露；关系修正能力通过 `add_relation`、`remove_relation`、`deprecate_relation` 接入 AI IDE；`cord init` 可以检测 IDE 和框架、生成配置与指令文件；Claude Code Hooks 与 Skills 生成把影响分析、关系查询、图谱初始化和同步建议纳入 Agent 工作流。

本 Epic 将前四个 Epic 的图谱能力带到用户日常开发入口。Epic 4 提供的 RelationService、`updateStrategy`、manual deprecated 保护语义在 Epic 5 中成为 MCP Tool 与 IDE 指令的业务基础；Epic 5 也为 Epic 6 的用户文档、MCP 参考文档和贡献者文档提供了实际可描述的产品面。

Sprint 状态中 Story 5.1-5.5 均为 `done`。本回顾完成后，`epic-5-retrospective` 标记为 `done`。需要注意的是，`epic-5` 主状态当前仍为 `in-progress`；这与所有 Story 已完成的事实不一致，建议由流程 owner 单独确认是否更新主 Epic 状态。

## What Went Well

1. MCP Server 从占位入口升级为真实 stdio 服务。Story 5.1 实现 `analyze_impact`、`query_relations`、`init_graph`、`sync_docs`，并补齐 stderr 日志、优雅退出、输入校验、并发只读调用与 schema 稳定性测试。
2. MCP I/O 契约被提升为架构规则。D13 / P44 / project-context 明确 CLI JSON DTO 是 MCP Tool 共享真源，避免 MCP 层自行裁剪字段或制造平行 DTO。
3. 关系管理 Tools 复用 Epic 4 的业务语义。Story 5.2 将新增关系建模为业务三元组，将删除和 deprecated 建模为 `relationId` 操作，并让 `query_relations` 输出稳定句柄，闭合 read-then-edit 使用链路。
4. IDE 适配器覆盖面完整。Story 5.3 实现 Claude Code、Cursor、VS Code Copilot、Codex CLI 四类适配器，并明确 `AGENTS.md` 是 NFR12 appendable 例外共享文件。
5. 零侵入策略经过 CR 加固。VS Code Copilot 的 `.github/copilot-instructions.md` 改为 protected write；`AGENTS.md` 已有 CORD 区块更新时保留区块外原始内容。
6. `cord init` 形成端到端编排。Story 5.4 编排 IDE 检测、框架检测、配置生成、数据目录创建、Skills 生成，并处理 `--format json`、`--ide` override、非 TTY 多 IDE 命中的 `AMBIGUOUS_IDE` 结构化错误。
7. Skills 生成从内联内容抽成独立模块。Story 5.5 新增 `skills-generator.ts`，并保持 `InitService -> IIdeAdapter.generateSkills?()` 编排链，让 FR31 的 owner 清楚落在 init 流程中。
8. Hooks 和三 IDE MCP 验证从“配置存在”升级为“可执行路径验证”。CR 最终确认 Hook 脚本实际调用 `cord impact --json`，三大 IDE 生成的 MCP 配置能通过 stdio 启动 server 并跑核心链路。
9. 验证纪律总体稳定。各 Story 记录了定向测试、全量测试、lint、type-check、build 或只读复核结论；阻塞项均在对应 CR 轮次闭合。

## Challenges

1. DTO 示例漂移仍未完全清理。TODO-034 记录 Story 5.1 历史 DTO 示例与当前 `src/mcp/tools/schemas.ts` 不一致，运行时代码已正确，但后续文档或 CR 回读旧示例时容易再次误导。
2. “共享文件但零侵入”是高复杂度边界。`AGENTS.md` 同时服务 VS Code Copilot 和 Codex CLI，必须既能 append，又不能破坏用户已有内容；这类例外需要规则文档、父 Epic、Story、测试一起同步。
3. `sync_docs` 边界经历多轮收敛。最终裁定为单文档只读建议 Tool，`action` 由 `updateStrategy` 推导，不写文档、不做自动编排；这个边界必须在 Epic 6 文档中继续保持清楚。
4. Init 输出摘要需要区分 generatedFiles 与 generatedSkills。Story 5.4 CR 发现重复摘要后已修复，说明用户可见结果字段也需要像 API DTO 一样被测试化。
5. Hooks 验证不能只停留在 settings JSON。Story 5.5 第一轮 CR 抓到执行性验证不足，后续补上脚本执行测试后才真正闭合 AC7。
6. 历史性能测试抖动仍影响全量测试叙事。QueryService 三跳性能阈值波动仍作为历史 CR TODO 跟踪，虽然不阻塞 Epic 5，但它反复出现在验证说明里，容易降低测试信号清晰度。
7. `cord init` 配置模板疑似未承接 `updateStrategies` 示例块。架构 D6 / P17 要求 init 生成配置模板包含 `updateStrategies` 示例；当前 `src/services/init-service.ts` 的 `buildInitConfig()` 与 `serializeYamlConfig()` 未写入该字段，init 相关测试也未覆盖该示例块。这是 Epic 4 交接项中最需要后续确认的缺口。
8. Epic 主状态收尾继续漂移。Epic 4 回顾已指出同类问题；当前 Epic 5 的所有 Story 已 done，但 `epic-5` 主状态仍是 `in-progress`。

## Key Insights

1. MCP Tool 不是单纯的薄包装；它是 CLI / Service DTO、schema 冻结、AI IDE 调用和用户文档之间的契约枢纽。
2. schema 稳定性必须同时冻结 input 和 output。只冻结 input 会漏掉 `relationId`、`hopDistance`、`durationMs`、`severity` 等下游消费必需字段。
3. 共享文件例外必须被显式命名。`AGENTS.md` 只有在 D9、P18、project-context、Story 和测试中全部同步，才能避免“零侵入”和“必须追加”互相打架。
4. 自动化初始化的用户体验来自细节组合：多 IDE 冲突、非 TTY JSON、配置格式、Skills 文件分组、Hooks 可执行性，每个边界都可能影响首次信任。
5. Epic 5 的 CR 模式证明：先用 Story Review 把契约收紧，再用 Code Review 检查真实生成物和执行路径，能有效捕获“文档说得对、实现没连上”的问题。
6. Epic 6 不应只写“功能清单”。它需要把 Epic 5 的真实边界写清楚：MCP Tool 字段、`sync_docs` 单文档只读语义、`query_relations.relationId` 句柄、`AGENTS.md` appendable 例外、init 生成物分类。
