## Epic Summary

Epic 1 的 5 个 Story 均已完成，交付了 CORD 后续功能开发所需的工程基座：分层 TypeScript/ESM 骨架、CordError 与 Logger、Zod 类型与 Schema、SQLite Repository 与迁移机制、GitHub Actions CI/CD 与质量门禁。

基于 Story 1.5 的收尾验证，Epic 1 结束时已具备完整质量门禁证据：`lint`、`type-check`、`build`、`test`、`coverage` 全部通过；最终本地回归记录为 200 个测试通过、整体覆盖率 98.35%。Epic 级 story review Round 5 结论为“通过”，无硬阻塞项。

## What Went Well

1. 工程骨架一次性搭建正确，后续 Story 都能沿同一层次边界扩展，没有出现架构方向回滚。
2. 共享契约收敛良好。CordError、Logger、RELATION_TYPES、Zod schema、Repository 接口在 Epic 1 内完成定型，为 Epic 2 的扫描、适配器、Service 直接提供依赖面。
3. 代码评审闭环有效。边界问题不是被口头接受，而是被转换成具体修复和回归测试，例如 CLI 入口符号链接与 `argv[1]` 缺失场景、query-input mixed-empty 输入、`relation_type` 数据库约束、release runtime 与权限边界。
4. 质量门禁执行到位。Epic 1 没有用“后续再补测试”绕过验收，所有核心 Story 都在 CR 收敛后形成稳定基线。

## Challenges

1. 真实风险主要集中在边界与运行时契约，而不是主干业务方向。CLI 入口、空字符串输入、数据库 schema 不变量、release workflow 运行时/权限边界都在后续 CR 中暴露出来。
2. 规则镜像文档同步存在过程性漂移。Epic 1 的 story review Round 5 明确指出，Story 1.5 已确认的 release 权限边界尚未完全回写到 Rule Document Registry 三份镜像文档；`RelationType` 术语和测试目录命名也仍有残留漂移。
3. 少量关键细节虽然最终实现了，但没有在最初任务拆分时被显式前置，例如 `--verbose` 接线时序、自动迁移触发点表达、PR 模板中的覆盖率验证命令。
4. Repository 与 release 流程中仍保留了明确的 pre-release TODO，这些不是当前阻塞项，但需要在扩大使用范围前主动处理，而不是继续隐含在 CR 文档中。

## Key Insights

1. Epic 1 已证明当前技术路线正确，主要风险不在“要不要这么做”，而在“边界条件是否被契约化并测试化”。
2. 每次高价值修复都伴随了新的定向回归测试，这是本 Epic 最值得保留的执行模式。
3. 文档和规则镜像同步不是行政工作，而是 AI Agent 能否稳定复用基线的重要组成部分；这类漂移如果不收口，会在后续 Epic 反复放大。
4. Epic 2 不需要重做规划。当前基座已经足够支撑扫描引擎、框架适配器和 ScanService 的实现，后续重点应放在复用既有契约与测试纪律。

## Technical Debt And Follow-up Backlog

1. 规则同步债务：将 Story 1.5 的 release 权限边界、`RelationType` 术语与测试目录命名同步回 Rule Document Registry 三份镜像文档。
2. Repository pre-release 债务：TODO-007 旧库升级路径与 SQL 迁移产物打包策略仍需明确，特别是在未来外部分发或真实持久库场景下。
3. Release 工程加固待办：release workflow 仍有“未依赖 CI 成功”“缺少 concurrency”“`[skip ci]` 条件过宽”“PR 模板未显式覆盖率命令”等非阻塞项。
4. 低优先级实现收口：`runCli` verbose 时序、路径/时间格式约束、partial update 过滤 `undefined`、mapper 错误统一类型等仍留在 CR TODO。

## Next Epic Preview

Epic 2 的目标是“文档扫描与关系图谱构建”。它对 Epic 1 的依赖非常直接：

1. `IFrameworkAdapter`、`AbstractFrameworkAdapter` 和 Generic/BMAD adapter 将直接建立在当前分层目录结构、门面导出和类型定义之上。
2. 扫描引擎需要复用 Epic 1 已定型的 `RELATION_TYPES`、Zod schema、CordError/Logger 和 CLI/MCP 约束。
3. `ScanService` 的冷启动扫描与图谱写入将建立在当前 SQLite Repository、事务模式、迁移机制和 sync state 结构之上。
4. `cord scan` CLI 薄壳需要沿用 Epic 1 已建立的 Commander 骨架、输出规范和错误处理流程。

结论：Epic 2 的方向无需改写，但应在启动前先把规则镜像一致性和少量 pre-release 决策收口，否则后续 Story 很容易在“读取旧规则”或“踩到已知边界”上重复返工。
