## Epic Summary

Epic 3 的 5 个 Story 均已完成，交付了 CORD 的读侧能力闭环：一跳关系查询、多跳 BFS 遍历、变更影响分析、JSON 快照导出和图谱健康检查。

本 Epic 将 Epic 2 建立的文档关系图谱转化为可消费能力。用户现在可以通过 `cord query` 理解单文档关系，通过 `cord impact` 获取下游影响和建议动作，通过 `cord export` 生成适合 git 审阅的快照，通过 `cord status` 查看图谱健康与配置状态。

Sprint 状态中 Story 3.1-3.5 均为 `done`。本回顾完成后，`epic-3-retrospective` 标记为 `done`。需要注意的是，`epic-3` 主状态当前仍为 `in-progress`；这与所有 Story 已完成的事实不一致，建议由流程 owner 单独确认是否更新主 Epic 状态。

## What Went Well

1. 读侧主链路完整交付。Story 3.1-3.5 覆盖查询、遍历、影响分析、导出、健康检查五个核心用户入口，并保持 CLI 薄壳 + Service 业务逻辑 + Repository 接口的架构边界。
2. QueryService 的语义在 CR 中显著变清晰。Story 3.2 修复了 `type` 过滤截断 BFS 路径的问题，并进一步修复了非匹配坏边在端点解析前阻断查询的问题；这直接沉淀为“可扩展边”和“可输出边”分离规则。
3. ImpactService 从通用查询复用转向自有传播语义。Story 3.3 通过 CR 修复了上游误报、低置信桥接边继续扩展、自环/回源环把源文档计入结果、重复命中膨胀 `totalCount` 等问题，最终形成有向三跳传播、阈值前置、按 impacted document 去重和 source self 排除的稳定语义。
4. ExportService 推动了配置契约补齐。Story 3.4 中 `project` 字段来源完成裁决，落地为 `cord.config.projectName` 优先、缺失回退项目根目录名，并同步更新 Rule Document Registry 三份文档。
5. StatusService 把“观测型命令必须只读”正式固化。Story 3.5 修复了未初始化项目执行 `cord status` 会创建 `.cord/cord.db` 的问题，并补充事务快照、dangling edge 统计和 cleanup best-effort 规则。
6. 测试纪律持续有效。每个 Story 都记录了窄测试、全量测试、lint、type-check 或 build 的通过证据；最终 Story 3.5 的 CR 摘要记录 `npm test`、`npm run lint`、`npm run build` 均通过。

## Challenges

1. 查询语义比初始 Story 描述更复杂。多跳查询不能简单把输出过滤当作遍历过滤；影响分析也不能简单复用通用双向查询再后过滤。这些都是通过 CR 才暴露的架构级细节。
2. 只读命令副作用治理不均衡。`status` 已修复为未初始化项目不创建存储，但 `query` / `impact` 默认 service 初始化仍由 TODO-028 跟踪。读侧命令的初始化语义需要统一收口。
3. 性能验收仍存在测试稳健性问题。Story 3.2 已补真实 SQLite repository 路径验证，但 TODO-024 仍指出 p95 比例测试受环境抖动影响，需要后续用索引命中断言或独立 benchmark 加固。
4. 导出快照的 git 审阅体验还有可靠性债。TODO-029、TODO-030、TODO-031、TODO-032 分别覆盖排序稳定性、原子写入、输出路径边界和配置错误路径副作用，当前均非阻塞，但会影响导出能力的长期可信度。
5. deprecated 关系的读侧语义尚未全局冻结。Query 和 Impact 已默认过滤 `status='deprecated'`；Export 应保留完整快照；Status 当前按全部关系统计，TODO-033 明确需要在 Epic 4 关系管理语义中单独裁决。
6. Sprint 状态口径继续出现漂移。Epic 3 的所有 Story 已完成，但 `epic-3` 主状态仍为 `in-progress`。这和 Epic 2 retro 中观察到的问题同类，说明 Epic 主状态收尾动作仍缺少稳定流程。

## Key Insights

1. 图查询功能的核心不是“能返回边”，而是“过滤、方向、深度、状态和坏数据处理都符合用户直觉且可解释”。
2. 对影响分析而言，关系边是传播路径，不只是展示对象。路径资格必须在扩展前判断，结果基数必须按受影响文档而不是关系命中数计算。
3. 观测型命令的价值来自可信边界：查看状态不能改变状态。Story 3.5 的修复应成为后续所有只读命令的模板。
4. JSON 快照是工程协作接口，不只是文件输出。排序稳定、覆盖语义、原子写入和路径边界会直接影响 git 审阅信任。
5. Rule Document Registry 在 Epic 3 中发挥了实际作用。新增 `projectName`、CR-QUERY、CR-STATUS 等规则都已同步到项目上下文和架构规则中，降低了后续 agent 重复踩坑的概率。
