## What Went Well

1. 关系管理语义完成闭环。Story 4.1 明确 `removeRelation` 是硬删除，`deprecateRelation` 使用 `status='deprecated'` 且保留原 `relationType`，手动添加固定写入 `source='manual'`、`status='active'`。
2. 迁移加固从单点修复上升为可复用规则。Story 4.1 补充 `relations.status` 增量迁移，并在 CR 后修复“列已存在但索引缺失”的部分迁移数据库场景，强化了 D2 / CR-REPO-06 的迁移子步骤独立幂等要求。
3. 收敛保护覆盖了真实用户路径。Story 4.2 修复后，用户手动 deprecated 原本由 `auto_scan` 或 `framework_preset` 发现的关系时，会转为 `source='manual'`，从而进入增量扫描保护集合。
4. 来源优先级在两个阶段都被测试化。Story 4.2 不只在持久化写入阶段处理 `manual > framework_preset > auto_scan`，也修复了同批次去重阶段按 confidence 误选 auto_scan 的问题。
5. rebuild 的数据保护边界清晰。`cord scan --rebuild` 在存在 manual 关系时会警告并确认；`--force` 可跳过确认但仍提示 manual 关系删除数量，符合 v0.1 对“全量重建一致性不包含 manual 状态”的裁决。
6. `updateStrategies` 与影响分析输出对齐。Story 4.3 新增 `auto | suggest | log_only`，默认回退 `suggest`，ImpactService 和 `impact` CLI 都输出 `updateStrategy`，同时同步 Rule Document Registry 三份文档。
7. CR 闭环质量高。Story 4.1 经过 3 轮、Story 4.2 和 4.3 经过 2 轮后均无剩余阻塞项或新增 CR TODO；每个 Story 都记录了定向测试、全量测试、lint 或 type-check 证据。

## Challenges

1. deprecated 语义容易被混用。Epic 4 一开始必须反复区分 `relationType` 与 `status='deprecated'`；如果把 deprecated 作为关系类型处理，扫描恢复、查询过滤和影响分析传播都会出现语义漂移。
2. ScanService 的保护逻辑存在多入口风险。source 文档删边、写入冲突、同批次去重、rebuild 删除都可能影响 manual 关系。最终方案覆盖了这些入口，但复杂度明显高于“扫描时保留 manual 边”这一句话。
3. 规则文档同步仍是高风险点。Story 4.3 第一轮 CR 发现 `updateStrategies` 旧语义残留，说明 Rule Document Registry 可以降低漂移，但仍需要审查流程持续约束。
4. `cord init` 模板契约在 Story 4.3 时只能文档化。Story 4.3 完成时 runtime `cord init` 尚未实现，因此 `updateStrategies` 示例块通过规则文档承接，后续实现必须再次验证模板输出没有漏掉该字段。
5. status 健康统计口径在回顾当日仍未冻结。TODO-033 当时仍为 open；后续已裁决为 active + deprecated 全量关系快照。
6. Epic 主状态收尾在回顾当日继续漂移。Epic 3 回顾已指出同类问题；当时 Epic 4 的所有 Story 已 done，但 `epic-4` 主状态仍是 `in-progress`。当前 sprint 状态已修正为 done。

## Key Insights

1. “用户修正”不是简单 CRUD，而是图谱收敛机制。手动修正必须影响后续扫描、查询、影响分析和 Agent 工具调用的行为。
2. deprecated 的正确建模是状态位，不是关系类型。这样才能让原始关系语义保留，同时让读侧和写侧按状态做一致决策。
3. 优先级规则必须覆盖扫描流水线中的每个候选淘汰点。只在最终写库时处理冲突，仍可能让高优先级关系在前置 dedupe 中被丢弃。
4. destructive 操作需要用户可见的保护边界。`--rebuild` 删除 manual 关系是合理的 v0.1 裁决，但必须在 CLI 层显式告知和确认。
5. 策略元数据与人读建议应并存。`updateStrategy` 服务机器编排，`suggestedAction` 服务人类理解，两者不能互相替代。
6. 文档规则同步本身需要测试和审查。规则文档不是“说明书”，而是后续 Agent 和 Story 的输入契约。
