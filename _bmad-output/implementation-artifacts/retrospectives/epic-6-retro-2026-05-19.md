## What Went Well

1. 文档交付范围完整。Epic 6 覆盖 README、快速开始、CLI 参考、MCP Tools 参考、配置参考、贡献指南和适配器指南，形成从首次使用到社区贡献的闭环。
2. MCP 文档承接了 Epic 5 的 schema 真源。`docs/mcp-tools-reference.md` 明确所有 inputSchema / outputSchema 都来自 `src/mcp/tools/schemas.ts`，并覆盖 7 个 Tool。
3. 关系管理闭环写清楚了。`query_relations` 输出中的 `relationId` 被明确说明为 `remove_relation` / `deprecate_relation` 的输入句柄，避免用户不知道如何从查询走到修正。
4. `sync_docs` 边界没有被扩大。文档明确它是单文档输入、只读同步建议，不执行文档写入，也不作为批量同步 facade。
5. 配置参考对实现差异保持诚实。`docs/configuration.md` 既给出完整配置模板，也说明当前 `cord init` 生成的基础配置范围，避免把未生成字段说成已自动生成。
6. `AGENTS.md` appendable 例外被写入用户文档。配置参考说明 CORD 只在 `<!-- CORD:START -->` / `<!-- CORD:END -->` 边界内追加或更新专属段，承接 D9 / P18 的零侵入例外规则。
7. 贡献者路径被文档化。`docs/adapter-guide.md` 覆盖 `IFrameworkAdapter` API、`AbstractFrameworkAdapter`、适配器注册、显式配置优先级、自动检测和 Generic 兜底链路。
8. Story 6.1 增加文档验收测试。`tests/unit/docs/adapter-contributor-docs.test.ts` 让贡献者文档的关键内容具备自动化保护，而不是只靠人工阅读。
9. CR 对文档质量的价值明确。Story 6.2 首轮发现两个 P1 Markdown 表格结构问题，修复后复审通过；这证明文档交付也需要结构化复核。
10. Epic 主状态收尾改善。Epic 4、Epic 5、Epic 6 主状态当前均为 `done`，此前回顾反复出现的主 Epic 状态漂移已被收口。

## Challenges

1. 文档中的表格语法是实际交付风险。Story 6.2 首轮 CR 发现 MCP outputSchema 与配置表格因为未转义管道符而错列，直接影响 schema 和配置项可读性。
2. 示例模板完整性仍有非阻塞缺口。Story 6.1 的贡献指南模板声明了重复扫描要求，但可复制模板还没有给出重复扫描断言；该项已进入 TODO-035。
3. 示例模板失败路径仍可增强。Story 6.1 的 SQLite 测试模板在断言失败时不保证关闭连接；该项已进入 TODO-036。
4. 历史 Story 示例漂移尚未清理。TODO-034 仍为 open，Story 5.1 的历史 DTO 示例仍可能误导后续读者或 CR 回读。
5. status 健康统计口径仍未冻结。TODO-033 仍为 open，deprecated 关系是否进入健康统计还需要产品/架构裁决。
6. `cord init` 的 `updateStrategies` 模板缺口没有在本 Epic 中修复。Epic 6 选择在配置文档中诚实说明当前 init 基础配置范围，但架构 D6 中“init 模板必须包含注释示例块”的规则仍与实现存在差异，需要后续裁决或修复。
7. README 徽章采用保守发布路径。Story 6.2 记录由于仓库未配置 git remote，Star / CI 徽章采用 `fancyliu/cord` 作为保守默认发布路径；发布前需要确认最终仓库坐标。

## Key Insights

1. 文档是契约消费层，不是实现完成后的附录。MCP schema、配置项、IDE 生成物和适配器扩展点都需要从源码真源映射到读者任务。
2. 文档示例需要测试化。只要文档里出现可复制代码或表格 schema，就应该有格式、字段和关键语义的自动化或脚本化检查。
3. 旧 Story 示例会继续产生影响。即使运行时代码已经修正，历史 Story 中过期 DTO 仍可能污染后续文档和评审，需要清理或显式标注。
4. 对实现差异保持诚实比“补齐叙事”更安全。`docs/configuration.md` 没有假称 `cord init` 已生成所有完整字段，而是把当前生成范围和可手动添加字段分开说明。
5. 社区贡献体验的关键不是只给接口，而是给激活链。贡献者需要知道在哪里注册、如何选择、如何检测、如何兜底、如何验证核心模块零修改。
6. Retrospective 的前置 action item 可以通过下一 Epic 直接吸收。Epic 5 要求 Epic 6 避免 MCP DTO 漂移、写清 `sync_docs` 和 `relationId`，这些都在 Story 6.2 中落地。
