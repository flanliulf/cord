## 发现 #2 评估

### 审查原文

> **[低] 集成测试模板在断言失败路径上不会关闭 SQLite 仓库连接**
> - 来源：edge
> - 分类：patch

### 评估结论：⚠️ 有效但降级 — 建议纳入 CR TODO 跟踪（P2 优先级）

### 评估分析

**问题描述准确性：准确**

`docs/contributing.md:57-58` 创建 `SqliteGraphRepository` 和 `ScanService`，模板断言集中在 `docs/contributing.md:60-65`，而 `service.close()` 位于 `docs/contributing.md:67`。如果任一 `expect(...)` 抛出失败，后续 `service.close()` 不会执行；`afterEach` 只删除临时目录（`docs/contributing.md:42-46`），没有集中关闭服务实例。

**严重性判断：合理但不阻塞**

原始严重性为“低”合理。该问题可能在断言失败或平台文件锁更严格时增加临时目录清理噪音，但它不影响文档是否包含贡献指南、PR 规范、审阅流程，也不影响当前实现运行时逻辑。作为模板健壮性问题，适合 P2 非阻塞跟踪。

**修复建议：可行但非必要**

将模板改为 `try { ... } finally { service.close(); }`，或在测试作用域跟踪 service 实例并由 `afterEach` 统一关闭，均可行。考虑到这是文档示例的失败路径加固，不要求本轮 fixer 立即处理。

**误报评估：非误报**

该发现与模板控制流一致，不是误报；但其交付风险较低，应降级为非阻塞改进。
