

Question:

在 AI CODING 的过程中，会逐渐产生很多文档，需要考虑定义一个 skills/mcp, 用 db 存储在 AI coding 过程中生成的项目文档之间的双链关系，供 AI 在参考某个文档时，能够通过该 skills 或者 MCP ，快速查询到该文档的相关联文档，从而能够快速修正其他文档对该文档的引用，或者对该文档修改内容的感知，或者添加对该文档的补充引用。
请分析上述需求是否合理，并给出建议。


---

Answer:

这个需求**非常合理，且极具前瞻性**。

在 AI 辅助开发的深度实践中，随着项目规模的扩大，最大的痛点往往不是“如何写出一段代码”，而是**“上下文的维护与一致性”**。AI 的上下文窗口是有限的，它无法每次都把整个项目的所有架构文档、API 定义和业务逻辑说明都加载到内存中。通过引入双链（Bi-directional Links）和 MCP（Model Context Protocol）来管理这些依赖关系，不仅切中了 AI Coding 的核心痛点，也是目前构建高级 AI 智能体（Agent）系统的标准演进方向。

特别是在处理复杂的 B2B 业务系统（例如涉及复杂的企微 API 对接、私域营销自动化流程、多租户架构等）时，一个底层数据结构或接口的改动，往往会牵扯到上层多个业务模块的逻辑更新。如果没有良好的文档关联机制，很容易出现“改了这头，忘了那头”的系统性腐化。

以下是对该需求的深度分析与落地建议：

### 一、 需求合理性分析

1.  **解决“上下文碎片化”问题**：通过双链图谱，AI 可以在需要时做到“按图索骥”，只拉取与当前任务强相关的文档，既节省了 Token，又提高了回复的准确度。
2.  **实现“级联更新”（Cascading Updates）**：当修改了核心文档（如某个 API 规范）时，AI 可以通过查询 DB 发现依赖该文档的其他模块说明，并自动提议或执行连锁更新，保持文档集的一致性。
3.  **高度契合 MCP 架构**：MCP 的核心设计初衷就是让大模型能够安全、标准化地与外部数据源（如数据库、本地文件系统）进行交互。将文档关系存储抽象为一个 MCP Server 是非常标准的最佳实践。



### 二、 架构与设计建议

为了让这个 MCP 真正实用，建议在设计时考虑以下几个维度：

#### 1. 存储方案的选择
* **轻量级方案（推荐起步）**：使用 **SQLite** 存储节点（文档）和边（引用关系）。建立一个类似于 `Links(source_doc_id, target_doc_id, relation_type, description)` 的表。对于中小型项目，这种方式配合 AI 已经足够高效。
* **专业级方案**：使用图数据库（如 **Neo4j** 或本地的轻量级图引擎）。图数据库天生适合处理这种双向网络，支持一键查询“某个文档的三度人脉（依赖链）”，非常适合分析复杂系统的改动影响范围（Impact Radius）。

#### 2. MCP 需要暴露的核心 Tool (能力)
你需要在这个 MCP Server 中定义以下几个核心工具供 AI 调用：
* `record_link(source_path, target_path, relation_type, context_summary)`: 记录引用关系。`relation_type` 可以是 `implements`, `depends_on`, `references`, `supersedes` 等。
* `get_linked_docs(file_path, direction="both", depth=1)`: 获取关联文档。支持查询正向依赖和反向被依赖的文档。
* `analyze_impact(file_path)`: 当需要修改某个文档时，一键拉取所有可能会受影响的下游文档列表及摘要。
* `remove_link(source_path, target_path)`: 在文档重构时清理废弃的链接。

#### 3. 关联粒度与元数据（Metadata）
* **文件级 vs 块级**：初期建议做到**文件级别**（File-level）的关联。未来如果需要更精细，可以参考 Obsidian 的设计，做到段落或标题级别（Block-level）的锚点链接。
* **引入“关系摘要”**：在记录 A 依赖 B 时，不要只存两个 ID，一定要让 AI 生成一句**摘要**存入 DB（例如：“营销触达模块依赖该文档定义的微信消息发送限制阈值”）。这样当 AI 查询依赖图谱时，光看摘要就能决定是否需要进一步拉取目标文档的全文。

### 三、 潜在挑战与避坑指南

1.  **数据状态不同步（Stale Data）**
    * **风险**：开发者手动删除了某个文档或修改了代码，但没有通过 AI 触发数据库的更新，导致数据库里存在“幽灵链接”。
    * **对策**：需要编写一个定期的自动化脚本（或在 CI/CD 流程中），扫描项目目录中的实际文件，并与 DB 中的记录进行校验（校验文件 Hash 或是否存在），清理失效链接。
2.  **上下文爆炸（Context Explosion）**
    * **风险**：如果一个核心组件被 50 个文档引用，`get_linked_docs` 一次性返回 50 个文档的全文会撑爆 AI 的上下文窗口。
    * **对策**：MCP 返回的内容必须是**分页的**或者**摘要聚合的**。AI 应该先拿到一个包含 50 个文档名称和简短摘要的列表，然后由 AI 自行决定进一步调用哪个文档的详细内容。
